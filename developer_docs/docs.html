1. System Overview

popy_tinkle operates as a split-architecture system:

  • The Browser Extension: A Manifest V3 extension built with TypeScript and Vite. It handles web request interception (via Declarative Net Request), metadata management, and user interfaces (Dashboard, Popup, Onboarding). It leverages the Origin Private File System (OPFS) for safe, localized storage.
  • The Native Daemon (popyd): A C++ system service that acts as a powerful companion to the extension. It manages cryptography (HMAC, keyrings), safe file system operations, file watching (fsevents for macOS, inotify for Linux), and sandboxed rendering (Markdown, PDF, Images).

2. Prerequisites

Before setting up the project, ensure you have the following installed on your development machine:

For the Browser Extension:

  • Node.js (v18 or higher recommended)
  • npm or yarn

For the Native Daemon (popyd):

  • A modern C++17/C++20 compatible compiler (Clang, GCC, or MSVC)
  • CMake (v3.15+)
  • Make or Ninja build system
  • Python 3.x (for running the MCP server hooks)

3. Local Development Setup

Browser Extension

The extension is located at the root of the repository.

  1. Install dependencies: 
    npm install
  2. Run the development server: 
    npm run dev

    This uses Vite to watch for file changes in src/ and output a hot-reloaded build.

  3. Load in the Browser:
    • Open Chrome/Edge and navigate to chrome://extensions/.
    • Enable "Developer mode" in the top right.
    • Click "Load unpacked" and select the generated dist/ directory (or wherever Vite is configured to output the build).

Native Daemon (popyd)

The C++ companion app is located in the /popyd directory.

  1. Configure the build with CMake: 
    cd popyd
mkdir build && cd build
cmake ..
  2. Compile the daemon: 
    make -j$(nproc)
  3. Configuration:

    Copy the example configuration file to your working directory before running:

    cp ../dist/config.example.toml ./config.toml
  4. Run locally (in foreground): 
    ./popyd --config ./config.toml

(Note: To install as a system service, see dist/com.popy.daemon.plist for macOS or dist/popyd.service for Linux).

5. Core Subsystems

Interception & Classification (Extension)

Web request interception is primarily handled via Chrome's Declarative Net Request (DNR) API. Logic in src/lib/interceptor/ dynamically evaluates host metadata (hostMatch.ts) and categorizes traffic (classifier.ts).

File Quarantine (popyd)

When untrusted files are downloaded or processed, popyd isolates them using src/store/quarantine.cpp. All file system operations must pass through src/core/safe_fs.cpp to prevent directory traversal attacks and ensure proper permissions.

Sandboxed Rendering (popyd)

Parsing external formats (PDF, Images, MD) is inherently risky. Code in src/render/ operates under strict security boundaries. If you are adding a new file format parser, you must implement it within src/render/sandbox.cpp.

6. Testing & Quality Assurance

We maintain strict testing requirements for both environments.

Extension Tests (Vitest)

Unit tests for the extension are located in /tests/.

# Run all TypeScript tests
npm run test

# Run tests in watch mode
npm run test:watch

Daemon Tests (CTest)

C++ tests are located in /popyd/tests/.

cd popyd/build
make test
# OR use CTest directly for verbose output
ctest --output-on-failure

7. Fuzzing

Because popyd handles untrusted inputs (MIME parsing, IPC messages, sidecar execution), we utilize fuzz testing. Fuzzers are located in popyd/tests/fuzz/.

To run the fuzzers locally (requires Clang with LibFuzzer):

  1. Configure CMake with fuzzing enabled (e.g., -DENABLE_FUZZING=ON).
  2. Build the fuzz targets: make fuzz_mime fuzz_attest fuzz_sidecar
  3. Execute the target: ./fuzz_mime

8. Continuous Integration (CI)

All pull requests must pass our automated GitHub Actions pipelines located in .github/workflows/:

  • ci.yml: Runs linters, Vitest (Extension), and CTest (Daemon).
  • fuzz.yml: Runs regression fuzzing on C++ targets.
  • release.yml: Automates packaging of the extension (.zip) and compiled binaries for tagged releases.

Please ensure your code compiles cleanly and passes all local tests before opening a Pull Request. For detailed PR guidelines, see CONTRIBUTING.md.
